## Step 1 — Security assessment

RustChain’s attestation path is a challenge-response flow intended to bind a miner wallet, miner identity, current node nonce, measured entropy, and hardware details into a report that the node can accept or reject. In [`rustchain-miner/src/attestation.rs` at `e1a48a0`](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rustchain-miner/src/attestation.rs#L199-L307), `attest_with_key` first requests a nonce from `/attest/challenge`, rejects an empty challenge, collects CPU timing entropy with `collect_entropy(48, 25000)`, builds a SHA-256 commitment from `nonce + wallet + entropy_json`, signs the critical tuple `miner_id|wallet|nonce|commitment` with Ed25519, and submits the assembled `AttestationReport` to `/attest/submit`. The report schema includes device metadata, network signals, optional fingerprint checks, miner version, public key, and signature ([same file lines 10-44](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rustchain-miner/src/attestation.rs#L10-L44)). This means the key security boundary is not just the JSON transport but the signed relationship between the wallet/miner identity and the nonce-bound commitment. If a verifier recomputes the same message and validates the signature, replay and wallet substitution become materially harder because a captured report is tied to a one-time nonce and to the signed wallet fields.

Hardware fingerprinting is used to raise the cost of VM farms and cloned miners. The miner gathers `HardwareInfo` in [`rustchain-miner/src/hardware.rs` at `e1a48a0`](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rustchain-miner/src/hardware.rs#L6-L115): platform, architecture, hostname, CPU family/architecture, core count, memory, platform serial where available, and MAC addresses. `generate_miner_id` hashes hostname plus serial into a short hardware-derived suffix ([lines 89-103](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rustchain-miner/src/hardware.rs#L89-L103)). At the consensus/design layer in [`rips/src/proof_of_antiquity.rs`](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rips/src/proof_of_antiquity.rs#L45-L58), submitted mining proofs are validated against hardware tier and multiplier expectations ([lines 145-203](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rips/src/proof_of_antiquity.rs#L145-L203)), then hashed using the model, generation, and `characteristics.unique_id` ([lines 322-335](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rips/src/proof_of_antiquity.rs#L322-L335)). The validator stores `known_hardware` and rejects the same hardware hash when it appears under a different wallet. There is also an `AntiEmulationVerifier` model with CPU signatures, cache ranges, and timing baselines ([lines 70-102 and 400-450](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rips/src/proof_of_antiquity.rs#L70-L102)), which is intended to compare submitted characteristics against real silicon behavior rather than accepting arbitrary self-declared hardware.

Epoch/block rewards are calculated after proofs are validated. [`ProofOfAntiquity::submit_proof`](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rips/src/proof_of_antiquity.rs#L145-L211) enforces a 120-second block window, rejects duplicate wallet submissions in the current block, checks persistent nonce reuse, caps block capacity at 100 miners, validates hardware age/tier/multiplier consistency, runs anti-emulation checks where characteristics exist, rejects hardware already registered to a different wallet, and caps multipliers at the Ancient-tier maximum. [`process_block`](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rips/src/proof_of_antiquity.rs#L213-L300) then sums all validated multipliers, assigns each miner `BLOCK_REWARD * (miner_multiplier / total_multipliers)`, records each `BlockMiner`, calculates a block hash from height, previous hash, total distributed reward, and timestamp, builds a Merkle root from miner entries, and resets pending proofs for the next window while deliberately keeping used nonces persistent. This reward design avoids simple “one proof gets all reward” behavior by making rewards proportional to the validated multiplier pool.

One attack vector I would prioritize is attestation/fingerprint spoofing through weak or non-persistent identity material. Some fields used by `HardwareInfo` can be absent, mutable, or easy to clone in virtualized environments: hostname, `/etc/machine-id` fallback, MAC addresses, and CPU brand strings are not hardware roots of trust. The code has compensating controls — nonce-bound Ed25519 signatures, duplicate hardware hash rejection, tier validation, timing/anti-emulation concepts, and persistent nonce tracking — but the verifier must ensure it never trusts client-supplied fingerprint pass/fail booleans without independently checking raw measurements and stable enrollment state. A VM farm could otherwise generate many plausible reports with unique hostnames/MACs, fresh signing keys, and self-selected timing metadata. The safest direction is to treat the client as untrusted, bind enrollment keys to a server-side hardware record, apply rate limits per hardware/network/wallet cluster, keep nonce/challenge state server-side with expiry, and require raw measurement distributions that are validated against architecture-specific baselines.

## Step 2 — Known fix reproduction: Mock Signature Mode

Selected BuilderFred audit class: **Mock Signature Mode**.

Before the fix, the dangerous pattern would be allowing a miner or test mode to submit attestation data without a real cryptographic binding between wallet, miner ID, challenge nonce, and commitment. In that failure mode, an attacker who can alter JSON in transit or generate synthetic reports could replace the `miner` wallet field, reuse a previous report with a fresh wrapper, or claim entropy/fingerprint results that were never signed by the enrolled miner key. The result is a classic authentication bypass: the node might see a structurally valid attestation object but have no proof that the wallet owner and the hardware report actually came from the same principal.

The fix is visible in [`rustchain-miner/src/attestation.rs` at `e1a48a0`](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rustchain-miner/src/attestation.rs#L10-L44). `AttestationReport` now carries both `signature` and `public_key`. `attest_with_key` computes the verifying key from the supplied signing key and rejects a mismatch with the provided public key ([lines 247-260](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rustchain-miner/src/attestation.rs#L247-L260)). Both `attest_with_key` and `attest` sign the exact message `miner_id|wallet|nonce|commitment` using Ed25519 ([lines 262-287 and 355-389](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rustchain-miner/src/attestation.rs#L262-L287)). The nonce comes from `/attest/challenge`; the commitment includes the nonce, wallet, and collected entropy. The report submitted to `/attest/submit` includes that signature and public key, giving the verifier the material needed to validate that any mutation of miner ID, wallet, nonce, or commitment invalidates the signature. The unit-test helpers in the same file also encode the intended verification behavior: sign a tuple, decode the public key and signature, and verify against the same message with `verify_strict` ([lines 423-493](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rustchain-miner/src/attestation.rs#L423-L493)).

This reproduction verifies the miner-side report construction and the in-repository signature verification tests; it does **not** prove that a production node endpoint currently enforces the same checks for every `/attest/submit` request. In the inspected tree, `/attest/submit` appears in the miner submission path, but I did not find a server-side submit handler that decodes `public_key`, rejects missing or malformed `signature` fields, recomputes `miner_id|wallet|nonce|commitment`, and calls Ed25519 verification before accepting the report. Therefore the Mock Signature Mode fix is sufficient only if that verifier exists in the deployed node or is added before accepting attestations. It also needs to ensure nonces are one-time-use and short-lived. The related consensus code in [`rips/src/proof_of_antiquity.rs`](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rips/src/proof_of_antiquity.rs#L157-L203) shows persistent nonce tracking in `used_nonces`, with a comment that nonce state is intentionally not cleared on block reset ([lines 296-300](https://github.com/Scottcjn/Rustchain/blob/e1a48a0f804d6160af62f24d0a92075ee47b9c26/rips/src/proof_of_antiquity.rs#L296-L300)), which is the right direction for replay resistance in the consensus proof path but is separate from HTTP attestation endpoint enforcement. Residual risk remains if the server accepts a fresh public key every time without an enrollment binding, because a bot can still create many independent keypairs; that is a Sybil/fingerprint problem rather than the original mock-signature bypass. The recommended hardening is to bind the first verified public key to a server-side miner/hardware enrollment record and require subsequent attestations for the same miner ID/hardware hash to verify under that enrolled key.
