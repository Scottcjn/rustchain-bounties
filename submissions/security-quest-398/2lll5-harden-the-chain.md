## Step 1 — Security assessment

RustChain’s attestation path is a challenge-response flow intended to bind a miner wallet, miner identity, current node nonce, measured entropy, and hardware details into a report that the node can accept or reject. In `rustchain-miner/src/attestation.rs`, `attest_with_key` first requests a nonce from `/attest/challenge`, rejects an empty challenge, collects CPU timing entropy with `collect_entropy(48, 25000)`, builds a SHA-256 commitment from `nonce + wallet + entropy_json`, signs the critical tuple `miner_id|wallet|nonce|commitment` with Ed25519, and submits the assembled `AttestationReport` to `/attest/submit`. The report includes device metadata, network signals, optional fingerprint checks, miner version, public key, and signature. This means the key security boundary is not just the JSON transport but the signed relationship between the wallet/miner identity and the nonce-bound commitment. If a verifier recomputes the same message and validates the signature, replay and wallet substitution become materially harder because a captured report is tied to a one-time nonce and to the signed wallet fields.

Hardware fingerprinting is used to raise the cost of VM farms and cloned miners. The miner gathers `HardwareInfo` in `rustchain-miner/src/hardware.rs`: platform, architecture, hostname, CPU family/architecture, core count, memory, platform serial where available, and MAC addresses. `generate_miner_id` hashes hostname plus serial into a short hardware-derived suffix. At the consensus/design layer in `rips/src/proof_of_antiquity.rs`, submitted mining proofs are validated against hardware tier and multiplier expectations, then hashed using the model, generation, and `characteristics.unique_id`. The validator stores `known_hardware` and rejects the same hardware hash when it appears under a different wallet. There is also an `AntiEmulationVerifier` model with CPU signatures, cache ranges, and timing baselines, which is intended to compare submitted characteristics against real silicon behavior rather than accepting arbitrary self-declared hardware.

Epoch/block rewards are calculated after proofs are validated. `ProofOfAntiquity::submit_proof` enforces a 120-second block window, rejects duplicate wallet submissions in the current block, checks persistent nonce reuse, caps block capacity at 100 miners, validates hardware age/tier/multiplier consistency, runs anti-emulation checks where characteristics exist, rejects hardware already registered to a different wallet, and caps multipliers at the Ancient-tier maximum. `process_block` then sums all validated multipliers, assigns each miner `BLOCK_REWARD * (miner_multiplier / total_multipliers)`, records each `BlockMiner`, calculates a block hash from height, previous hash, total distributed reward, and timestamp, builds a Merkle root from miner entries, and resets pending proofs for the next window while deliberately keeping used nonces persistent. This reward design avoids simple “one proof gets all reward” behavior by making rewards proportional to the validated multiplier pool.

One attack vector I would prioritize is attestation/fingerprint spoofing through weak or non-persistent identity material. Some fields used by `HardwareInfo` can be absent, mutable, or easy to clone in virtualized environments: hostname, `/etc/machine-id` fallback, MAC addresses, and CPU brand strings are not hardware roots of trust. The code has compensating controls — nonce-bound Ed25519 signatures, duplicate hardware hash rejection, tier validation, timing/anti-emulation concepts, and persistent nonce tracking — but the verifier must ensure it never trusts client-supplied fingerprint pass/fail booleans without independently checking raw measurements and stable enrollment state. A VM farm could otherwise generate many plausible reports with unique hostnames/MACs, fresh signing keys, and self-selected timing metadata. The safest direction is to treat the client as untrusted, bind enrollment keys to a server-side hardware record, apply rate limits per hardware/network/wallet cluster, keep nonce/challenge state server-side with expiry, and require raw measurement distributions that are validated against architecture-specific baselines.

## Step 2 — Known fix reproduction: Mock Signature Mode

Selected BuilderFred audit class: **Mock Signature Mode**.

Before the fix, the dangerous pattern would be allowing a miner or test mode to submit attestation data without a real cryptographic binding between wallet, miner ID, challenge nonce, and commitment. In that failure mode, an attacker who can alter JSON in transit or generate synthetic reports could replace the `miner` wallet field, reuse a previous report with a fresh wrapper, or claim entropy/fingerprint results that were never signed by the enrolled miner key. The result is a classic authentication bypass: the node might see a structurally valid attestation object but have no proof that the wallet owner and the hardware report actually came from the same principal.

The fix is visible in `rustchain-miner/src/attestation.rs`. `AttestationReport` now carries both `signature` and `public_key`. `attest_with_key` computes the verifying key from the supplied signing key and rejects a mismatch with the provided public key. Both `attest_with_key` and `attest` sign the exact message `miner_id|wallet|nonce|commitment` using Ed25519. The nonce comes from `/attest/challenge`; the commitment includes the nonce, wallet, and collected entropy. The report submitted to `/attest/submit` includes that signature and public key, giving the verifier the material needed to validate that any mutation of miner ID, wallet, nonce, or commitment invalidates the signature. The unit-test helpers in the same file also encode the intended verification behavior: sign a tuple, decode the public key and signature, and verify against the same message.

This is sufficient for the specific Mock Signature Mode class if the node enforces verification on every `/attest/submit` request and rejects missing, malformed, or unverifiable signatures. It also needs to ensure nonces are one-time-use and short-lived. The related consensus code in `rips/src/proof_of_antiquity.rs` shows persistent nonce tracking in `used_nonces`, with a comment that nonce state is intentionally not cleared on block reset, which is the right direction for replay resistance. Residual risk remains if the server accepts a fresh public key every time without an enrollment binding, because a bot can still create many independent keypairs; that is a Sybil/fingerprint problem rather than the original mock-signature bypass. The recommended hardening is to bind the first verified public key to a server-side miner/hardware enrollment record and require subsequent attestations for the same miner ID/hardware hash to verify under that enrolled key.
